# RecallLayer Repositioning Plan

## Purpose

This plan repositions the project from an experimental standalone "TurboQuant-native vector database" framing toward a more credible and implementable product direction:

> **RecallLayer is a vector retrieval sidecar for existing databases.**

That means the project should be described less as a full database replacement and more as a focused retrieval engine that:
- stores compressed vector indexes
- supports hybrid retrieval over mutable and sealed state
- returns candidate ids and scores for rerank or hydration
- integrates with an existing system of record such as Postgres

## Positioning statement

### One-line version

**RecallLayer is a compressed vector retrieval sidecar for existing databases.**

### Short product version

**RecallLayer is a vector retrieval engine that plugs into your existing stack. It handles compressed indexing, hybrid search, and rerank-ready candidate generation while your primary database remains the source of truth.**

### Honest current-state version

**RecallLayer is an early but increasingly disciplined retrieval engine focused on compressed vector search, storage lifecycle correctness, and sidecar-style integration with existing systems.**

## Why this positioning is stronger

The current repository already has meaningful engine work around:
- mutable + sealed lifecycle
- flush semantics
- recovery replay boundaries
- compaction / retirement / garbage collection
- benchmark and proof flows
- local and API query surfaces

What it does **not** yet have is the broader shape of a production-ready standalone database product:
- full durability / checkpointing story
- mature multi-tenant operational model
- broad security / auth / admin surface
- distributed architecture
- source-of-truth application storage model

Because of that, the strongest believable product story is:
- **existing DB owns the records**
- **RecallLayer owns vector retrieval**

## Product category

RecallLayer should be positioned as one or more of the following:
- vector retrieval sidecar
- compressed vector index service
- rerank-ready candidate generation layer
- hybrid retrieval engine for existing systems

Avoid leading with:
- standalone database replacement
- production-ready vector database platform
- full pgvector replacement

## Ideal use cases

### 1. RAG / document retrieval sidecar
- source records live in Postgres or another application database
- embeddings are generated by the app or an upstream pipeline
- RecallLayer stores compressed vector search state
- query returns candidate ids + scores
- app hydrates content from source DB

### 2. Semantic search over large corpora
- knowledge bases
- docs search
- support archives
- internal enterprise search

### 3. Recommendation / candidate generation
- similar items
- content recommendation
- feed candidate generation
- ad/content matching

## Core product promise

RecallLayer should promise:
- lower-memory vector retrieval than keeping all search state as full-precision vectors in memory
- explicit retrieval lifecycle behavior
- hybrid retrieval over mutable + sealed state
- candidate generation that fits a rerank architecture
- easier adoption because the primary database does not need to be replaced

RecallLayer should **not** yet promise:
- complete database replacement
- full production durability guarantees
- broad distributed deployment maturity
- universal algorithmic leadership claims

## Recommended technical milestone

### Deliverables
1. Rebrand core docs around RecallLayer
2. Define the integration contract
3. Define canonical Postgres + RecallLayer sidecar architecture
4. Add at least one black-box integration example

   - insert
   - update
   - delete
   - flush
   - restart
   - recovery
   - candidate-id hydration expectations
6. Clarify current guarantees and non-guarantees

## Canonical interface recommendation

The project should support both:
- **embedded Python library** for local development and experiments
- **HTTP sidecar API** as the canonical real-world integration story

Why this combination works:
- library mode keeps iteration fast
- HTTP sidecar mode matches how real apps would integrate
- both can share the same storage lifecycle semantics

## Suggested acceptance criteria for repositioning

Repositioning is successful when:
- a new reader can understand RecallLayer in one paragraph
- the project no longer sounds like it is claiming to replace a primary database
- the sidecar architecture is explicit
- the source-of-truth boundary is explicit
- product and technical language no longer depend on Google's terminology as the primary framing
